Effort Estimation — Concepts & Techniques

Every project plan has estimates at its heart. And every estimator lies — not on purpose, but because estimation is genuinely hard. This week we understand why, and learn techniques that reduce the error.

01 · Why estimation is hard

Nobody's being dishonest — it really is this difficult10 min

Ask someone how long it takes to drive from London to Edinburgh. They'll say about 7 hours. They've done it before, or something similar. There's a reference class of prior experience to draw on.

Now ask how long it takes to build a new feature in a legacy codebase that uses a framework nobody on the team has touched before. The reference class barely exists. The person estimating is making an educated guess about something they've never quite done, in a context that's slightly different from every prior project, involving people they may not know well yet.

That's the fundamental problem with software estimation. It's not a math problem — it's a novelty problem. The more novel the work, the less accurate the estimate.

The Cone of Uncertainty — estimate error narrows as the project progresses

The implication of the Cone: at the start of a project, the best possible estimate has a 4× error range. You might think something will take 100 days; the real answer could be anywhere from 25 to 400. This isn't an excuse — it's a constraint that planning must account for. The plan must communicate uncertainty honestly, and be revisable as that uncertainty narrows.

Why estimates are usually too low
Optimism bias
Humans systematically underestimate how long things take — especially novel things. We imagine the best-case scenario, not the expected-case scenario.
The planning fallacy
Even knowing that similar projects ran late, estimators believe "this one will be different." Daniel Kahneman's research on this is extensive — and humbling.
Anchoring
If someone asks "can we do this in 3 months?" the answer often becomes "yes, with effort" — even if the honest estimate is 6 months. The initial number anchors the negotiation.
Invisible work
Estimates typically capture coding time. They miss: meetings, code review, bug fixing from prior sprints, onboarding new team members, waiting for decisions, documentation. These routinely add 50–100% to raw development estimates.
02 · Estimation approaches

Four ways to arrive at an estimate20 min

Top-down
Estimate the whole, then decompose
Use a high-level model or expert judgment to estimate the total effort, then allocate portions to phases and activities. Fast, but relies heavily on the accuracy of the top-level model. Good for: early feasibility estimates when detail isn't available yet.
Bottom-up
Estimate each task, then sum
Break the work into the smallest manageable tasks, estimate each individually, then add them up. More accurate for well-understood work. Requires a complete WBS first — which you may not have early in a project. Good for: detailed planning once scope is established.
Analogy
Compare to a similar completed project
Find a completed project that resembles this one. Adjust the estimate based on the differences. Requires good historical data — which most organizations don't track carefully enough. COCOMO II (Week 5b) uses this approach algorithmically.
Expert judgment
Ask someone who's done it before
Can be highly accurate for familiar work. The risk is overconfidence in the expert's judgment and anchoring on their first number. The Delphi technique (below) is a structured way to get expert estimates without anchoring bias.
Delphi estimation

Gather a group of experts. Each person writes their estimate privately (no discussion first). Reveal all estimates simultaneously. Discuss the outliers — why did one person estimate 5 days when another said 20? The person with the high estimate often knows something the others don't. Repeat until consensus emerges. The structure prevents anchoring and surfaces hidden knowledge.

Planning poker (used in agile) is a version of Delphi — cards are revealed simultaneously so nobody anchors on the first number spoken.

03 · Function Point Analysis

Measure what the software does, not how it's built20 min

Lines of code is a terrible size measure — 1000 lines in Python might do the same thing as 5000 lines in Java. Function Points (Albrecht, 1979) measure what the software does for the user, regardless of how it's implemented. This makes estimates comparable across languages, platforms, and teams.

There are five types of function to count. Each occurrence is weighted by complexity (Simple / Average / Complex). Sum the weights to get the Unadjusted Function Point Count (UFC):

Internal Logical Files (ILF)71015Data maintained by the system (databases, files the app owns)

Rough effort: ~0 person-hours (at 8 hrs/FP average)

After the UFC: apply a Value Adjustment Factor (VAF) — 14 general system characteristics (performance requirements, online data entry, transaction rates, etc.) rated 0–5. The VAF adjusts the UFC by ±35%: FP = UFC × (0.65 + 0.01 × Σ ratings). The adjusted FP count can then be converted to hours using historical data from similar projects — typically 8–20 person-hours per function point depending on the domain.

04 · Wrap-up

Key takeaways

All estimates are wrong — the question is by how much
Don't aim for perfect precision. Aim for honest uncertainty ranges. A plan that says "this will take 12 weeks ± 3 weeks" is more useful than one that says "12 weeks" with false confidence.
Communicate uncertainty explicitly
The Cone of Uncertainty should be shown to stakeholders, not hidden from them. A project that delivers in month 5 against an early estimate of "month 3–6" is a success. A project that delivers in month 5 against a committed deadline of month 3 is a failure — even if the estimate was always unrealistic.
Function Points separate size from technology
Measuring in function points lets you compare estimates across teams, languages, and time. Build an organizational database of historical FP data — it's one of the most valuable assets a PM organization can have.
